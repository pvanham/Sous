## Phase 1: Security, Identity, and Context Boundaries
Objective: Ensure the AI only acts within the exact permissions of the user and understands the spatial context of the UI without trusting it blindly.

Dynamic Role-Based Access Control (RBAC) Tool Filtering:

The Concept: The orchestrator must intercept the user's session token before building the LLM context. It evaluates the user's role (e.g., General Manager vs. Shift Lead) and dynamically prunes the allowed tools list. A Shift Lead physically cannot ask the AI to modify financial parameters because the AI is not provided the update_cost_weights tool in its payload.

Zero-Trust Viewport Injection:

The Concept: The frontend client constantly streams its current "Viewport Context" (e.g., active week ID, selected staff member) alongside user messages.

The Guardrail: The backend orchestrator must treat this Viewport Context strictly as conversational context, not authorization. If the frontend claims the user is looking at "Restaurant A," the backend must independently verify the user has access to "Restaurant A" before executing any database queries.

## Phase 2: The Bounded Tool Registry (The "Pull" Architecture)
Objective: Equip the LLM with highly specific data-gathering capabilities that prevent context window bloat and prompt injection.

Strict Parameter Schematization:

The Concept: Every tool (Read and Write) must be defined with rigid input schemas (using a validation library). The LLM must be explicitly told the exact format required to trigger a function.

Data Minimization & Aggregation Layer (Read Tools):

The Concept: Tools designed to fetch data (like schedule health or shift rosters) must never return raw database rows. The backend execution layer must aggregate and format the data into concise, token-efficient JSON summaries before handing it back to the LLM.

Pagination Enforcements: Any tool capable of returning lists must have hard limits (e.g., max 20 records) and return pagination metadata to prevent the LLM from accidentally requesting a year's worth of shift history.

Prompt Injection Sanitization:

The Concept: Any data fetched by a Read Tool that contains user-generated text (like staff time-off notes) must be wrapped in strict XML delimiters (e.g., <untrusted_user_text>) before entering the LLM's context. The system prompt must instruct the LLM to never execute commands found within these tags.

## Phase 3: The Generative UI & Human-in-the-Loop (HITL) Circuit
Objective: Detach the AI's intention to mutate data from the actual execution of that mutation, ensuring zero accidental data destruction.

The Proposal Stream:

The Concept: When the LLM calls a "Write Tool" (e.g., propose_shift_swap), the orchestrator does not execute a database query. Instead, it streams a structured payload to the frontend, which renders an interactive Confirmation Card UI component directly in the chat history.

Optimistic Concurrency Control (OCC):

The Concept: To prevent the "Stale Card" pitfall, every Confirmation Card generated by the AI must include a hidden version hash or timestamp of the underlying data at the moment the card was created.

The Execution: When the user clicks "Approve," the backend compares the card's version hash against the current database version. If another manager altered the schedule while the card was sitting in the chat, the backend rejects the execution and prompts the user to refresh.

Component State Hydration & Expiration:

The Concept: The frontend must manage the lifecycle of these interactive cards. Once a card is approved, denied, or if it belongs to a previous chat session, the interactive buttons must be programmatically disabled to prevent users from accidentally clicking old actions.

## Phase 4: Asynchronous Task Orchestration
Objective: Seamlessly integrate the AI conversational flow with long-running, deterministic compute tasks (like the constraint programming solver) without triggering serverless timeouts.

The "Fire and Forget" Handoff:

The Concept: Tools that trigger heavy compute (like generating a new weekly schedule) must be entirely asynchronous. The tool dispatches the payload to the external compute microservice and immediately returns a "Task Started" status to the LLM.

The Follow-Up Trigger (System Wake-Up):

The Concept: The LLM goes idle while the external compute service runs. The frontend must listen for the compute service's completion (via polling or webhooks).

The Execution: Upon completion, the frontend silently dispatches a programmatic system message to the orchestrator (e.g., [SYSTEM: Solver completed successfully. Total cost: $4,200.]). This invisible prompt wakes the LLM up, provides the results, and forces it to generate a natural language summary for the user, keeping the conversational UX fluid.

Graceful Failure Handling:

The Concept: If the external compute service returns an "Infeasible" or "Failed" status, the follow-up trigger must pass that exact error state to the LLM so it can empathize with the user and suggest relaxing specific constraints (like the overtime limits or clopening rules).

## Phase 5: Telemetry, Observability, and Audit Logging
Objective: Create an immutable paper trail of every action the AI proposes and the user approves, which is a hard requirement for enterprise workforce management software.

The AI Audit Trail:

The Concept: A dedicated database schema must log every successfully executed "Write Tool."

The Execution: The log must capture the User ID who clicked approve, the specific Tool invoked, the exact JSON payload executed, and the Chat Session ID. This ensures that if a schedule change violates a labor law, management can trace exactly how the decision was formulated and approved.

Graceful Step-Limit Failures:

The Concept: If the orchestrator hits its maximum allowed consecutive tool calls (preventing an infinite loop), it must not crash silently. It must log the loop failure in the telemetry system and append a final system instruction forcing the AI to apologize to the user and request clarification.
